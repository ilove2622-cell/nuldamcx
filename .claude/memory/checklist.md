# 작업 검증 체크리스트 & 실수 기록

## 과거 실수 사례

### 1. 코드 작성 후 미검증 (게시판, 채팅상담, 클레임)
- **증상**: 파일 생성 후 "완료"라고 보고했지만 빌드 에러, 런타임 에러 발생
- **원인**: `npm run build` 안 돌림, API route 실제 호출 안 해봄
- **교훈**: 코드 작성 ≠ 완료. 반드시 빌드 + 동작 확인 후 보고

### 2. 다중 파일 수정 시 정합성 깨짐 (클레임 시트 열 확장)
- **증상**: 시트 열을 A~J → A~N으로 바꿨는데 일부 파일만 반영
- **원인**: 영향받는 파일 목록을 사전에 파악 안 함
- **교훈**: 스키마/구조 변경 시 관련 파일 전체 목록을 먼저 나열하고 하나씩 대조

### 3. 환경변수 누락 상태에서 배포 (AICC, 클레임)
- **증상**: 배포 성공이지만 502/500 에러
- **원인**: 코드에서 참조하는 환경변수가 플랫폼에 미설정
- **교훈**: 새 환경변수 추가 시 즉시 "어디에 설정해야 하는지" 명시

### 4. 에러 발생 시 근본 원인 분석 없이 코드 수정
- **증상**: 한 곳 고치면 다른 곳이 깨지는 연쇄 문제
- **원인**: 에러 메시지를 정확히 읽지 않고 추측으로 수정
- **교훈**: 에러 → 로그 확인 → 원인 특정 → 최소 범위 수정

### 5. 배포 후 실제 URL 미확인
- **증상**: 배포는 됐는데 실제 접속하면 안 됨
- **원인**: 빌드 통과 ≠ 정상 동작
- **교훈**: 배포 후 반드시 실제 URL로 핵심 기능 1개 이상 호출 확인

## 작업 유형별 필수 검증

### API Route 생성/수정
1. [ ] 코드 작성
2. [ ] `npm run build` 통과 확인
3. [ ] curl 또는 브라우저로 실제 호출 → 응답 확인
4. [ ] 에러 케이스(잘못된 파라미터 등) 테스트
5. [ ] 필요한 환경변수 목록 정리 → 로컬/.env + Vercel + Railway 설정 확인

### 프론트엔드 페이지 생성/수정
1. [ ] 코드 작성
2. [ ] `npm run build` 통과 확인
3. [ ] 의존하는 API route가 정상 동작하는지 먼저 확인
4. [ ] 페이지 접속 → 핵심 기능 동작 확인

### 다중 파일 수정 (스키마/구조 변경)
1. [ ] 변경 전: 영향받는 파일 전체 목록 나열
2. [ ] 파일별 수정
3. [ ] 수정 후: 모든 관련 파일 다시 읽어서 필드명/타입/순서 대조
4. [ ] `npm run build` 통과 확인

### 배포
1. [ ] 로컬 빌드 성공 확인
2. [ ] Railway: `git push origin main`
3. [ ] Vercel: `npx vercel --prod`
4. [ ] 배포 후 실제 URL로 API 1개 이상 호출 확인
5. [ ] 새 환경변수가 있으면 플랫폼별 설정 확인

## 작업 원칙

1. **작은 단위로 나눠서 진행**: 1~2개 파일 작성 → 검증 → 다음 파일
2. **API 먼저, UI 나중에**: API route가 실제 응답하는 걸 확인한 뒤 프론트엔드 작업
3. **"됐다"고 말하기 전에 증거 확보**: 빌드 로그, API 응답, 스크린샷 등
4. **에러는 추측 금지**: 에러 메시지 → 원인 특정 → 최소 수정
5. **외부 의존성은 별도 관리**: 환경변수, 서비스 계정, 버킷 등 수동 작업 목록 유지
